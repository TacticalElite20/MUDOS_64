MUDOS_64, meaning Modern Ultra Deterministic Operating System based on 64-bit, is designed as a small, modern desktop operating system that can be improved gradually without requiring its basic architecture to be replaced. Its main objective is to combine the directness and low overhead of older operating systems with selected modern capabilities such as 64-bit memory protection, multicore scheduling, hardware-enforced process isolation, USB support, graphical desktop operation and IOMMU-based DMA protection. The design deliberately avoids trying to provide every feature expected from Windows, Linux, macOS or FreeBSD. Instead, it establishes strict limits, a narrow hardware target and a relatively small set of clearly defined behaviours so that the complete system remains understandable and maintainable by a solo developer.

The operating system is completely UI-centric. The desktop environment is not treated as a separate optional graphical program running above the operating system, but as an integrated part of the system itself. When the computer starts, it boots directly into the desktop without presenting a login screen, user account selector or profile system. MUDOS_64 is strictly a single-user operating system. It does not implement multiple users, file ownership, user permissions or separate administrative accounts. The absence of these systems reduces complexity, although it also means that the system is not intended to protect files belonging to one human user from another human user.

MUDOS_64 uses a fully monolithic kernel. The kernel, scheduler, desktop environment, window manager, compositor, filesystem, process manager, device drivers and operating-system API all run together in the most privileged processor mode. They are internally divided into separately organised areas and execution loops, but they remain parts of one integrated kernel rather than independent protected services. This follows a philosophy similar to FreeRTOS, in which the complete system is monolithic but individual responsibilities are divided internally. There is no separate initialization process because the kernel itself performs initialization and permanently represents the core system. The kernel is identified as process zero, while separately loaded applications run as ordinary external processes.

This architecture allows the kernel’s components to communicate directly and efficiently. The UI can interact directly with the filesystem, the scheduler can directly manage process objects and drivers can deliver processed input directly into the UI event system. There is no need to send every operation through multiple protected service processes. The trade-off is that an error in any privileged component can potentially damage the complete system. A faulty display driver, filesystem routine, scheduler operation or window-manager routine can therefore cause a kernel panic. The design accepts this risk and follows the principle that a properly designed monolithic system should not contain kernel defects that need to be isolated from the rest of the kernel.

The system is intended only for 64-bit x86 desktop computers using UEFI firmware. It does not support legacy BIOS booting, 32-bit processors or alternative processor architectures. It may happen to run on laptops, but laptops are not an intentional design target and the system is not initially designed around laptop-specific power management, battery reporting, lid switches, touchpads or hybrid graphics. The narrow platform target reduces the number of compatibility layers and hardware abstractions that need to be maintained.

The boot process starts in UEFI. The firmware loads the operating system and provides access to the Graphics Output Protocol framebuffer, allowing MUDOS_64 to display graphics before a native GPU display path has been fully initialized. The kernel obtains the system memory map, identifies available and reserved physical memory, initializes its physical page allocator and establishes its own virtual-memory mappings. It then initializes exception and interrupt handling, configures the local interrupt controller timer, detects supported floating-point and vector instruction sets, initializes the processor privilege structures, creates the scheduler, initializes the desktop environment and finally enables normal interrupt-driven operation. Once the kernel has obtained the information it needs from UEFI, it properly exits UEFI boot services so that the operating system no longer depends on firmware-controlled memory management or ordinary firmware services.

The default display resolution is 2560 by 1440. The graphical output is scaled to occupy the complete display from corner to corner. Initially, graphical drawing is performed mainly by the CPU. Changed desktop and application areas are rendered and then passed through the graphical output path into video memory before being sent to the monitor. AMD Display Core is intended to provide the core display-management functionality, but not the complete functionality of a modern accelerated graphics driver. The operating system therefore does not initially depend on 3D acceleration, complex GPU command submission or a full modern graphics stack.

The visual style is a heavily simplified but intentionally accurate interpretation of the Windows NT 4.0 desktop. Windows, menus and controls use flat two-dimensional graphics, hard edges and manually drawn shapes. There are no animations. The compositor is limited to basic screen scaling, transparency and a simple grey shadow effect. Applications and system components provide colour values using a conventional human-readable red, green, blue and alpha range, while the compositor internally converts the colours into the form needed for efficient alpha blending.

The desktop environment uses invalidation-based repainting. Instead of redrawing the entire screen continuously, the system marks only the regions that have changed. Examples include an area exposed by moving a window, a cursor position, a changed line of text, a selected icon or a newly opened menu. The system supports a large fixed number of invalidation regions so that repainting can remain precise while the checks for changed areas remain inexpensive. Repainting is event-driven rather than tied to an unrestricted constant full-screen redraw loop.

The desktop contains a fixed grid of 25 columns and 13 rows for positioning icons. Detected application files appear as white rectangular desktop icons with their exact filenames displayed underneath. If a name is too wide to fit within its grid position, it is shortened and ends with an ellipsis. Icons can be selected with one left click, opened with a double left click and moved by dragging. A selected icon receives a translucent blue highlight over its complete icon area. Files can also be moved into folders by dragging them onto folder icons.

Application windows can be moved and resized by dragging their borders. Each window has a title on the left side of its title bar and only three controls on the right: minimize, fullscreen and close. A minimized program remains available on the taskbar and can be restored by clicking its taskbar entry. Fullscreen mode can be exited with the Escape key. Closing a window terminates the corresponding process through the normal process-death mechanism rather than merely hiding the window.

The window manager maintains a simple ordered stack of windows and clips drawing so that applications cannot visually overwrite areas belonging to windows in front of them. Bringing a window to the foreground moves it forward in this order. The design intentionally avoids complex animated transitions, virtual desktops, window effects and extensive shell extensions.

The Start menu is a simple grey rectangle. It displays the current MUDOS_64 version at the bottom as non-interactive text. Above this are shutdown and reboot options. The top of the menu can display the three most recently launched application programs, with the most recently opened program shown first. At boot, before any applications have been opened, these recent-program entries are absent rather than being shown as empty placeholders.

A right mouse click opens the Fast Menu, which functions as the desktop context menu. It provides operations for copying, pasting, renaming, creating folders and permanently deleting files. Copying creates a completely independent duplicate of the selected file rather than a shortcut or second reference to the original file. A copied application is therefore independently runnable and can be launched at the same time as the original. Deletion is immediate and permanent, with no recycle bin or undo mechanism.

The input path is deliberately separated from immediate graphical work. Keyboard and mouse interrupts collect hardware input and place it into a fixed-size input queue. The driver-processing logic interprets these queued events and sets the appropriate UI events. The desktop environment then responds, marks the necessary graphical regions as invalid and redraws those areas. Interrupt handlers do not directly redraw windows or perform expensive UI processing. This avoids visible stuttering and prevents long graphical operations from blocking other interrupts.

All normal keyboard keys are supported. Advanced mice with two additional side buttons are also supported. The main system keyboard shortcuts include Control and Escape for shutdown, Control, Alt and Escape for reboot, the Windows key for toggling the Start menu and Control, Alt, Shift and Escape for closing the foreground application.

MUDOS_64 uses a static timer tick of exactly 1,000 ticks per second, meaning that the main hardware timer interrupts the system every millisecond. This timer drives scheduling, process wake-ups, time measurement and other periodic operating-system activities. Logical time is kept separate from rendering delay and temporary system lag. As a result, timed behaviour such as a blinking cursor remains based on actual elapsed time instead of slowing down permanently when a frame is delayed.

The system supports up to 128 external application processes in addition to the kernel itself. Each application launch creates a separate process, including when the same application is opened multiple times. Two instances of the same program therefore have independent memory, execution state, windows and process identities. Process identifiers are reused by choosing the lowest available number, but each process also receives a separate instance identity. This prevents an old reference to a terminated process from accidentally referring to a new process that later receives the same numerical process identifier.

Applications can be running, ready to run, blocked while sleeping, migrating between processor cores or dead. A running process is actively executing on a logical processor. A ready process is able to execute but is waiting for its turn. A blocked process is temporarily sleeping and is excluded from scheduling until its requested wake-up time is reached. The timer checks sleeping processes and returns them to the ready state at the correct time. When no runnable work exists on a logical processor, that processor enters a hardware idle state and waits for an interrupt instead of wasting power by repeatedly checking for work.

A process selected for movement between processor cores temporarily enters a migration state. It is first stopped on its current processor, removed safely from the old run queue and then transferred to the target processor’s queue. Its previous state is preserved so that, for example, a sleeping process remains sleeping after migration. Termination has priority over migration, ensuring that a process marked for death cannot remain alive simply because it was being moved between cores.

A dead process is no longer scheduled. If it was running when termination was requested, the scheduler switches away from it before completing its destruction. The process object remains present only as long as another part of the kernel still holds a valid reference to it. Once no references remain, its identifier, memory mappings, stacks, extended processor state and other resources can be released.

The scheduler uses a round-robin model with a preferred scheduling latency of 64 milliseconds and a minimum timeslice of two milliseconds. With only a few runnable processes, each process receives a relatively large share of the 64-millisecond target period. As the number of runnable processes increases, the timeslices become shorter, but they never fall below two milliseconds. Because of this minimum, a complete rotation through many processes may intentionally take longer than the nominal 64-millisecond target. This avoids excessive context-switch overhead when a large number of processes are active.

A process switch saves the complete ordinary execution state of the outgoing process as well as its floating-point and vector state. If the next process uses a different address space, the active page-table hierarchy is changed. The processor’s privileged stack information is updated for the new process, its saved state is restored and execution returns to user mode. This allows programs to use x87 floating point, SSE, SSE2, AVX, AVX2 and FMA3 without corrupting each other’s calculations.

The scheduler uses separate run queues for each logical CPU instead of one universal queue. This reduces contention because each processor can usually manage its own scheduling independently. Processes may be migrated when one logical processor contains at least two more runnable tasks than another. After migration, the process must remain on its new processor for at least approximately 64 milliseconds before another migration is allowed. This prevents frequent movement between processors, which would otherwise damage cache locality and increase locking overhead.

On processors with separate performance and efficiency cores, MUDOS_64 groups them into a performance-core pool and an efficiency-core pool. Work is initially concentrated on the performance-core pool. The efficiency-core pool becomes available when average performance-core utilization reaches approximately 90 percent. Once enabled, it remains active until performance-core utilization stays below approximately 70 percent for around two seconds. This separation prevents the efficiency cores from being repeatedly enabled and disabled when utilization fluctuates near one threshold. The policy is primarily intended to preserve performance and responsiveness rather than to achieve the lowest possible power use.

CPU utilization is measured as the proportion of time each logical processor was busy over the preceding 64 milliseconds. Pool utilization is calculated as the average of the logical processors belonging to that pool. Scheduler load balancing uses the number of ready and currently running processes rather than attempting to predict the exact computational intensity of each process.

Shared scheduler and memory-management structures are protected using spinlocks, atomic operations and explicit memory-ordering barriers. Data used only by one processor can be updated while local interrupts are disabled, preventing interrupt handlers on that processor from observing an incomplete state. Disabling local interrupts does not protect information shared with other processors, so shared structures still require proper synchronization. When two processor run queues must be locked simultaneously, the queue belonging to the lower-numbered processor is always locked first. This fixed ordering prevents two processors from deadlocking while attempting opposite migrations.

Memory management uses four-level 64-bit virtual memory with fixed 4 KiB pages. Physical memory is tracked through a bitmap in which each bit represents the availability of one physical page. Small kernel allocations are handled using fixed-size allocation groups ranging from 16 bytes to 2,048 bytes. Each group divides one or more physical pages into equal-sized objects and records which objects are free. Requests larger than 2,048 bytes are allocated directly using page-granular physical memory.

This approach provides fast and predictable handling of commonly sized kernel objects while limiting the behaviour of general-purpose heap fragmentation. It does introduce some internal waste because an allocation is rounded up to the next available size class. The design intentionally accepts this bounded waste in exchange for simple and reliable allocation behaviour.

Every external process has its own complete page-table hierarchy. One process cannot normally read or modify the private memory of another process because their address spaces are separated and enforced by the processor’s memory-management unit. Kernel memory remains privileged and inaccessible from Ring 3. Programs execute only in Ring 3 and enter the kernel through the official MUDOS_64 API.

Memory is allocated lazily. A process begins with a small initial stack, and additional stack pages are supplied only when the process actually touches valid stack-growth addresses. The stack may grow to a maximum of 384 KiB. The heap begins with no committed memory and expands only when the application requests and uses it. This avoids reserving large amounts of physical memory for processes that never need it.

If a new process cannot receive the memory required to start, the process is simply not launched. If an already running process requests additional valid memory and the system cannot provide it, that process is terminated. If the kernel itself can no longer obtain memory and no external applications remain whose termination could recover sufficient space, the system displays a kernel out-of-memory panic.

Address-space layout randomization is used so that application images are not always loaded at identical virtual addresses. The executable format stores its entry point relative to the randomly chosen image base. This makes memory addresses less predictable and provides a limited security improvement against attacks that depend on fixed code locations.

Shared memory is fully supported for efficient communication and data sharing between processes. A shared physical page can be mapped into the address spaces of multiple programs, potentially with different access permissions. The system tracks both the shared object and every process-specific mapping. Shared-memory objects and process mappings use instance identities in addition to reusable numerical identifiers, preventing stale references from attaching to newly created objects.

MUDOS_64 does not implement threads. When an application needs parallel execution, the preferred model is to launch multiple processes and coordinate them through shared memory and optimized interprocess communication. This is heavier than threads because each process has its own page tables, process state and privileged stack, but it keeps the execution model simpler and gives each parallel task a stronger memory boundary. Because the total number of external processes is limited to 128, this process-based parallelism remains bounded.

The native executable format is called MEXE version 1.5. It is a little-endian x86-64 format intended only for Ring 3 user programs. It describes the program’s code, initialized data, uninitialized data and entry position. Programs begin with a 4 KiB stack that grows lazily up to 384 KiB, while their heap begins empty and grows on demand. Applications are built separately from the operating system and loaded dynamically from the filesystem.

Applications communicate with MUDOS_64 through the published system API. The kernel and other core system files do not contain hardcoded locations for specific applications. Programs can therefore be added, removed, renamed, replaced or copied without rebuilding the operating system. Applications may themselves request system operations, create windows, access files, use shared memory, launch other programs and communicate through the approved API. This modular application model is one of the main boundaries between the fixed operating-system core and software developed later.

The native filesystem is intentionally compact. It supports files, directories and mounted volumes within one virtual namespace. Each object has a stable entry identity, a parent identity, a volume identity, a name, a size, a single timestamp and a type. The timestamp is obtained from the real-time clock and records the year, month, day, hour, minute and second. The size is stored internally as an exact byte count and displayed dynamically in an appropriate unit such as bytes, kilobytes, megabytes or gigabytes.

The filesystem supports at most 32,768 entries. Name lookup uses a fixed-size indexed hash structure with twice as many lookup slots as maximum entries, keeping the table at no more than half capacity. This is intended to provide fast and predictable average lookup performance while avoiding long lookup chains. Allocation of file data and free space is tracked through bitmaps. If insufficient storage space exists for a write, the write is rejected rather than partially completed or silently overcommitted.

The native filesystem is case-sensitive. Files with names differing only by capitalization can therefore be treated as separate objects. Names are stored using UTF-8 with up to 48 encoded bytes. Because some Unicode characters occupy more than one byte, this does not always correspond to 48 visible characters. External USB filesystems preserve the capitalization shown by the original filesystem but compare names according to that filesystem’s own semantics, which are generally case-insensitive for FAT32, exFAT and typical NTFS usage.

The core system uses a global root directory from which files, folders and volumes can be accessed. Mounted USB storage devices appear as volume objects. USB drives are detected automatically by periodically checking for newly connected mass-storage devices. The USB mass-storage implementation understands FAT32, exFAT and NTFS. For a mounted drive, the displayed timestamp represents when the drive was mounted, while its displayed size represents total capacity.

The File Manager presents filesystem contents as a simple textual list in a black window. Every item displays its name, timestamp and size. Directories and mounted drives can be entered, and the left arrow returns to the parent directory. At the root, pressing left has no effect. The selected item is surrounded by a blue rectangular cursor similar to an older firmware menu. Pressing Enter or the right arrow opens the selected object.

Application files are launched, text files are opened in the Text Editor, folders are entered and mounted volumes display their contents. When an unsupported file type is selected, a global notification window asks whether the user wants to attempt opening it as plain text. Accepting opens the file in the Text Editor with its contents interpreted as text, while rejecting closes the notification without performing another action.

The notification window is a single global grey window with centred black text. Applications and system components can supply the message to be shown. This avoids every program having to implement its own error, warning and notification window.

The Terminal uses a black background and a square cursor that blinks every 400 milliseconds. It supports commands for permanently deleting a named file, launching a named application, killing a process by identifier, listing mounted disks, shutting down and rebooting. The command set is intentionally narrow and does not attempt to provide a Unix-style shell, scripting language, pipes or complex command expansion.

The Text Editor also uses a black background and square blinking cursor. When creating a new file, it first asks for a filename and uses F1 to confirm the name. The user can then write text and press F1 to save. When an existing text file is opened through the File Manager, the editor receives that file directly and does not ask for a new name.

Pong is a conventional Pong game controlled with the up and down arrow keys. Minesweeper uses a fixed 16 by 16 grid, with left click used to reveal tiles and right click used to mark suspected mines. The Clock is a simple digital display based on the real-time clock and shows the current time in hours, minutes and seconds.

The Task Manager uses a black graph with a rolling 30-second history sampled every 50 milliseconds. It displays total logical CPU utilization, memory usage, the number of active processes relative to the system limit, memory fragmentation and disk fragmentation. Each quantity has its own line colour and matching numeric label. The process graph treats the complete global limit, including the kernel, as 100 percent.

The Stress Test application is designed to place extreme pressure on the scheduler and memory system. It uses a fixed 12 GiB working set, intensive 64-bit arithmetic, pseudo-random memory updates, page-stride accesses and operations that cannot be optimized away by the compiler. It updates its statistics every 100 milliseconds and reports average throughput, minimum and maximum throughput, variability, high-percentile short-term changes and the worst observed 100-millisecond stall. Its purpose is not merely to produce a high performance number, but to expose scheduler unfairness, memory bottlenecks, latency spikes, page-allocation issues and multicore balancing problems.

The audio system uses a deliberately fixed format. Audio is signed 16-bit little-endian stereo PCM at 44.1 kHz. System output volume is fixed at 50 percent. Applications are responsible for decoding compressed or containerized audio formats and sending decoded PCM to the operating system. The driver itself does not interpret MP3, WAV or other application-level file formats. Small fixed audio buffers are used to keep latency low. During startup, MUDOS_64 produces a 1,180 Hz sine tone for two seconds, after which it remains silent until an application requests audio output.

The initial driver set covers framebuffer output, keyboards, mice, the system timer, the real-time clock, shutdown and reboot handling, USB human-interface devices, audio and memory management. The broader driver layout reserves space for Bluetooth, Wi-Fi, Ethernet, SATA, NVMe, USB audio, USB mass storage, the shared USB core and AMD display support. Some of these sections are initially empty and serve as planned expansion points.

The operating system uses Intel VT-d and the IOMMU to restrict direct memory access by PCI devices. Every active DMA-capable requester receives its own protection domain wherever the hardware permits independent isolation. A device initially has access to no normal system memory. Only the buffers explicitly registered for that device are mapped into its domain, and each buffer receives only the read or write access that the hardware operation actually requires.

PCI bus mastering remains disabled until the protection domain and required mappings are active. During device removal, shutdown or reset, bus mastering is disabled before the mappings and physical buffers are released. Outstanding DMA operations are completed or stopped before their memory is reclaimed. If a device attempts unauthorized DMA, the IOMMU blocks the access, records the requester and attempted address and disables the responsible device. The system produces an IOMMU DMA violation panic only when safe operation cannot continue.

Software security is intentionally limited. The primary protections are Ring 3 execution for applications, separate virtual memory for every process, memory permissions enforced by the processor, address randomization, validation through the system-call interface and IOMMU protection against unauthorized device access. The design does not include antivirus scanning, multiple security principals, elaborate application sandboxes or extensive background security software. Its security strategy depends heavily on a small attack surface and strict validation at the limited number of interfaces exposed to applications and devices.

The kernel includes a watchdog intended to detect a system that has failed to make progress for more than 30 seconds. A fatal condition displays a red kernel-panic screen with a large white title and a smaller explanation. Defined panic categories include a general Ring 0 execution failure, an unrecoverable IOMMU DMA violation, a watchdog timeout, a kernel-level out-of-memory condition and a processor double fault. A double fault represents an exception that occurred while the processor was already attempting to handle another exception and is treated as a severe unrecoverable failure.

The operating system’s implementation is organised into a small fixed set of source areas covering the kernel, scheduler, drivers, UI, assembly routines, application API, executable linker, executable-building tool, built-in applications and build script. The source is visually divided into large functional sections, with system-wide variables and tunable values placed together so that behaviour can be adjusted easily. The intended code style avoids excessive commentary inside implementation sections and prioritizes compactness and immediate visual recognition of the code’s purpose.

The overall design is strongly bounded. It has fixed limits for processes, filesystem entries, filename length, UI invalidation regions, input events, desktop icon positions, stack size, scheduler timing and other resources. These limits make memory consumption and failure behaviour easier to understand. When a capacity is reached, the corresponding operation is expected to fail explicitly rather than causing uncontrolled growth.

MUDOS_64 is therefore best understood as a compact graphical operating system with a modern protected process model, multicore execution, virtual memory, USB storage, a native application format, direct desktop integration and strong DMA isolation, while deliberately excluding the broad compatibility and administrative features of larger systems. Its complexity is concentrated in areas that are required for modern hardware operation, such as paging, processor-state switching, SMP synchronization, storage support, display output and IOMMU configuration. It avoids spending comparable complexity on user accounts, permissions, services, compatibility layers, threads, visual effects and automated background behaviour.

The result is intended to feel like a modernized operating system from an earlier era: immediate boot into a complete desktop, direct control over applications and files, very little hidden behaviour, fixed and understandable limits, a small native software model and a kernel whose major components remain visible as one integrated system. Its central engineering goal is not to reproduce every modern operating-system feature, but to provide the greatest possible amount of useful desktop functionality while preserving simplicity, predictable scaling, low overhead and long-term maintainability.
